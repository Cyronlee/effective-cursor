## AI 辅助 Spec 框架简介

AI 辅助 Spec 框架代表了软件开发范式的重大转变，核心理念是 **Spec-Driven Development（SDD，规范驱动开发）**。在这种方法中，开发者优先创建详细、可执行的规范（包括需求、设计、任务等），再由 AI 辅助生成、完善和实现代码。这种方式解决了传统“vibe coding”（凭感觉随意提示）或临时提示导致的 AI 输出不可预测、偏离意图等问题。

通过将规范作为“真相之源”（source of truth），这些框架显著提升了协作性、可审计性、代码质量和开发效率，尤其适用于复杂项目中的人机协作开发。

## 三大主要框架概览

### OpenSpec
由 Fission-AI 团队开发的开源框架，专为 AI 编码助手设计的 SDD 工具。  
核心特点是变更管理（change management），特别适合**已有项目（brownfield）**的迭代开发。  
工作流程：将每个功能变更组织在单独文件夹中（`openspec/changes/feature-name/`），包含提案（proposal）、任务（tasks）和规范增量（spec deltas），采用类似 patch 的方式更新。  
AI 作用：自动生成提案、迭代完善规范、指导实现、归档已批准变更。本地运行，无需 API key，支持 20+ 种 AI 工具（通过 slash 命令集成）。  
优势：变更追踪清晰、可审计，适合跨多个文件/规范的修改。

### Spec-Kit
GitHub 官方推出的开源工具包，旨在帮助开发者快速上手可执行规范驱动开发。  
核心特点：**可执行规范**（executable specifications），规范本身可直接驱动代码生成。  
提供 CLI 工具 `specify`，支持项目初始化、模板管理和 slash 命令（如 `/speckit.specify`、`/speckit.plan`、`/speckit.tasks`）。  
工作流程：规范 → 澄清 → 规划 → 任务拆分 → 实现，支持多种 AI 编码助手（Claude、Gemini、Copilot 等）。  
强调技术无关性、企业约束和用户导向，适合**全新项目（greenfield）**和**已有项目**。

### Kiro Spec
AWS 推出的 **Kiro**（一个基于 VS Code 的 Agentic AI IDE）内置的规范系统。  
核心理念：**结果导向而非实现细节**，终结混乱的“vibe coding”，让开发更结构化、可控。  
三阶段工作流：  
1. **Requirements**（需求）：生成带详细验收标准的用户故事  
2. **Design**（设计）：产出架构、图表、schema 等  
3. **Implementation**（实现）：拆分成可跟踪的任务列表  
Kiro 支持“vibe 模式”（对话式）和“spec 模式”（规范驱动），可从提示/原型自动生成规范，支持多模态输入（代码、图片、文档等），并通过 Agent Hooks 实现自动化（如保存文件自动更新测试/文档）。  
优势：深度代码库理解、原型到生产的全流程支持，特别适合快速原型和复杂功能开发。

## 共同点

三大框架共享以下核心特征，共同定义了 AI 时代 SDD 的本质：

- **以规范为核心**：规范是“真相之源”，代码只是规范的一种表达形式  
- **AI 深度参与**：利用 AI 自动生成、完善、执行规范，减少人工重复劳动  
- **结构化分阶段流程**：从意图 → 规范 → 设计 → 任务 → 实现，形成闭环反馈  
- **多 AI 工具兼容**：支持主流 AI 编码助手（如 Claude、Copilot、Gemini 等）  
- **目标一致**：提升软件质量、可维护性、可审计性，减少误解和返工，加速可靠交付

## 区别对比

| 维度             | OpenSpec                              | Spec-Kit                              | Kiro Spec                                   |
|------------------|---------------------------------------|---------------------------------------|---------------------------------------------|
| **主要定位**     | 变更管理 + 已有项目迭代（brownfield） | 快速启动工具包（greenfield & brownfield） | 全功能 Agentic IDE（原型 → 生产）           |
| **工作流风格**   | 基于变更文件夹 + delta 增量更新       | CLI + slash 命令分阶段（specify → plan → tasks） | 三阶段明确分离（Requirements → Design → Tasks） |
| **AI 集成深度**  | 本地、无 API key、20+ 工具 slash 命令 | 工具无关，多 AI 代理支持              | 深度代码库分析、自动规范生成、Agent Hooks 自动化 |
| **集成方式**     | 独立框架，嵌入现有项目                | CLI 工具 + 项目模板                   | 内置于 VS Code 风格的专用 IDE               |
| **独特优势**     | 变更追踪清晰、跨规范修改友好          | 可执行规范 + 项目宪法（constitution） | 多模态输入、自动更新规范、原型快速验证      |
| **适用场景**     | 长期维护、大型已有系统                | 快速实验、新项目启动               | 复杂功能、团队协作、从 0 到生产全流程       |

## 与传统手动维护规范的区别

AI 辅助 Spec 框架与传统手动编写/维护 Word、Wiki、Notion 等规范文档有本质不同：

- **效率**：AI 自动从提示/原型/代码库生成规范，迭代只需几分钟；手动可能耗时数小时甚至数天  
- **准确性与一致性**：AI 深度理解代码上下文，减少歧义、遗漏和前后矛盾；人工规范极易出现理解偏差和过时  
- **动态性**：规范可实时更新（Kiro 甚至保存文件自动刷新），成为“活文档”；传统规范通常静态，容易与代码脱节  
- **可追溯性**：变更历史清晰（OpenSpec delta、Kiro 版本控制），便于审计和 review；手动文档追踪困难  
- **可执行性**：规范直接驱动代码生成与验证，成为开发闭环核心；传统规范仅作参考，需人工解读  
- **规模化能力**：轻松应对复杂、大型项目；手动维护随项目增长迅速变得不可持续

**总结一句话**：  
AI 辅助 Spec 框架将规范从**静态文档**转变为**动态、可执行、可自动化的真相之源**，真正实现了“先想清楚，再让 AI 帮你实现”，这是 2025-2026 年 AI 辅助软件工程最重要的发展方向之一。
